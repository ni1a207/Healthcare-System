## Назначение

Определяет ролевую модель системы: какие роли существуют, какие права каждая роль имеет и как `api-gateway` принимает решение о доступе на основе роли из JWT. FR-000 является основополагающим для всех остальных FR — каждый функциональный блок системы опирается на роли, описанные здесь.

---

## AS-IS процесс

**Описание текущего состояния:**

В таблице `users` (`auth-service-db`) поле `role` имеет тип `VARCHAR(50)` без ограничений на допустимые значения. Фактически в системе существуют две роли: `USER` (основная) и `ADMIN` (тестовый аккаунт в `data.sql`). `auth-service` включает `role` в JWT claim при генерации токена, однако `api-gateway` этот claim не читает и не использует — проверяется только валидность подписи JWT. Любой авторизованный пользователь вне зависимости от роли имеет доступ ко всем эндпоинтам системы.

**Известные проблемы AS-IS:**

- Нет разграничения доступа между сотрудником, врачом и пациентом — любой авторизованный пользователь может выполнять любые действия
- `api-gateway` игнорирует claim `role` из JWT — маршрутизация не зависит от роли
- Отсутствуют роли `DOCTOR` и `PATIENT` — невозможно разграничить доступ к медицинским данным
- Нет ограничений на допустимые значения поля `role` в БД

## TO-BE процесс

**Описание целевого поведения:**

В системе вводятся три роли. Роль пользователя включается в JWT claim `role` при аутентификации. `api-gateway` при каждом запросе выполняет две последовательные проверки: валидацию JWT и проверку роли. Эндпоинт `GET /validate` обновляется — при успешной валидации возвращает роль и email пользователя в заголовках ответа, на основании которых `api-gateway` принимает решение о доступе.

**Роли системы:**

| Роль | Кому назначается | Описание |
|------|-----------------|----------|
| `ADMIN` | Сотрудник регистратуры / администратор клиники | Полный доступ к управлению пациентами, расписанием, пользователями |
| `DOCTOR` | Врач клиники | Доступ к расписанию своих приёмов, карточкам пациентов, проведению приёмов |
| `PATIENT` | Пациент клиники | Доступ к личному кабинету, самозаписи, оплате |

## Алгоритм работы

1. `auth-service` через `DispatcherServlet` принимает входящий HTTP-запрос `GET /validate`.

2. `DispatcherServlet` вызывает метод `validateToken(authHeader)` в `AuthController`. Spring извлекает значение заголовка `Authorization` через аннотацию `@RequestHeader("Authorization")` и передаёт его в параметр `authHeader`. Если заголовок `Authorization` отсутствует в запросе — Spring автоматически возвращает HTTP статус-код `400 BAD REQUEST` до выполнения кода контроллера.

3. `AuthController` проверяет что значение `authHeader` начинается с префикса `Bearer `. Если префикс отсутствует — `AuthController` возвращает HTTP статус-код `401 UNAUTHORIZED`.

4. `AuthController` отсекает префикс `Bearer ` (первые 7 символов) через `authHeader.substring(7)` и вызывает метод `authService.validateToken(token)`.

5. `AuthService` вызывает метод `jwtUtil.validateToken(token)` внутри блока try-catch. `JwtUtil` с помощью библиотеки `JJWT` выполняет последовательные проверки токена:
   5.1. Проверка структуры — токен должен состоять из трёх частей (`Header.Payload.Signature`), разделённых точками.
   5.2. Проверка подписи — `secretKey` используется для вычисления хеша и сравнения с `Signature` в токене. При несовпадении — `JJWT` выбрасывает `SignatureException`, который перехватывается и оборачивается в `JwtException("Invalid JWT signature")`.
   5.3. Проверка срока действия — из `Payload` извлекается клейм `exp` и сравнивается с текущим временем сервера. Если токен истёк — `JJWT` выбрасывает `JwtException`.

6. Если `jwtUtil.validateToken(token)` выбросил любое исключение наследуемое от `JwtException` — `AuthService` перехватывает его в блоке catch и возвращает false. `AuthController` получает false и возвращает HTTP статус-код `401 UNAUTHORIZED`.

7. Если все проверки пройдены успешно — `jwtUtil.validateToken(token)` завершается без исключения, `AuthService` возвращает true. `AuthController` получает true.

8. 🟢 `AuthController` извлекает из JWT claims: `role` → заголовок `X-User-Role`, `sub` (email) → заголовок `X-User-Email`.

9. 🟢 `DispatcherServlet` отправляет клиенту `200 OK` с заголовками `X-User-Role: {role}` и `X-User-Email: {email}` без тела ответа.

Если в процессе выполнения возникает критический сбой — необработанное исключение не связанное с валидацией токена (например, ошибка конфигурации секретного ключа) — `auth-service` возвращает HTTP статус-код `500 INTERNAL SERVER ERROR`.

## Общая информация

**Назначение:** 🟢 Цепочка двух фильтров на всех защищённых маршрутах. `JwtValidation` проверяет подпись и срок действия JWT, извлекает роль и email пользователя из ответа `auth-service` и пробрасывает их в запрос. `RoleValidation` проверяет что роль пользователя входит в список разрешённых ролей для текущего маршрута.

## Алгоритм работы при запросе к защищённым ресурсам системы

1. `api-gateway` принимает HTTP-запрос на порту 4004. Компонент `Route Configuration` сопоставляет URI запроса с правилами маршрутизации в `application.yml`. Для маршрута в списке `filters` указаны `JwtValidation` и 🟢 `RoleValidation`.

2. `Route Configuration` передаёт управление фильтру `JwtValidationGatewayFilterFactory`.

3. `JwtValidationGatewayFilterFactory` извлекает значение заголовка `Authorization` из входящего запроса. Если заголовок отсутствует или не начинается с префикса `Bearer ` — `api-gateway` немедленно возвращает клиенту HTTP статус-код `401 UNAUTHORIZED`, запрос не проксируется.

4. `JwtValidationGatewayFilterFactory` через реактивный `WebClient` инициирует HTTP-запрос `GET /validate` к `auth-service` с заголовком `Authorization: Bearer <token>`.

5. `auth-service` выполняет валидацию токена и возвращает ответ. Если `auth-service` недоступен или вернул любой не-2xx статус — `WebClient` выбрасывает `WebClientResponseException`, которое не перехватывается — `api-gateway` возвращает клиенту HTTP статус-код `500 INTERNAL SERVER ERROR`.

6. 🟢 При получении `200 OK` от `auth-service` — `JwtValidationGatewayFilterFactory` извлекает заголовки `X-User-Role` и `X-User-Email` из ответа и добавляет их к входящему запросу. Управление передаётся следующему фильтру — `RoleValidationGatewayFilterFactory`.

7. 🟢 `RoleValidationGatewayFilterFactory` читает заголовок `X-User-Role` и сравнивает с `allowedRoles` текущего маршрута из `application.yml`. Если роль не входит в `allowedRoles` — `api-gateway` возвращает клиенту `403 FORBIDDEN`, запрос не проксируется.

8. 🟢 Если роль разрешена — `RoleValidationGatewayFilterFactory` передаёт управление следующему фильтру в цепочке `chain.filter(exchange)`. `api-gateway` проксирует запрос к целевому сервису. Префикс `/api` удаляется через `StripPrefix=1` — запрос `/api/patients/123` преобразуется в `/patients/123`.

9. `api-gateway` получает ответ от целевого сервиса. Если сервис недоступен — возвращается HTTP статус-код `500 INTERNAL SERVER ERROR`.

10. `api-gateway` проксирует ответ клиенту.

- [ ] Роль `USER` выведена из использования, существующие записи мигрированы в `ADMIN`
- [ ] Поле `role` в таблице `users` ограничено допустимыми значениями: `ADMIN`, `DOCTOR`, `PATIENT`

**api-gateway:**
- [ ] Фильтр `JwtValidation` обновлён — пробрасывает `X-User-Role` и `X-User-Email` в проксируемый запрос
- [ ] Фильтр `RoleValidation` реализован
- [ ] Все маршруты настроены с указанием `allowedRoles` согласно матрице доступа
- [ ] При запрещённой роли возвращается `403 FORBIDDEN`

- [ ] Интеграционный тест: `ADMIN` → `POST /api/patients` → `201 Created`
- [ ] Интеграционный тест: `DOCTOR` → `POST /api/patients` → `403 Forbidden`
- [ ] Интеграционный тест: `PATIENT` → `GET /api/patients` → `403 Forbidden`
- [ ] Интеграционный тест: невалидный JWT → любой защищённый маршрут → `401 Unauthorized`
