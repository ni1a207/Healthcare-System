## Назначение

Сотрудник клиники регистрирует нового пациента в системе через модальную форму. После создания пациент получает на email приветственное письмо с логином, сгенерированным паролем и ссылкой на личный кабинет.

---

## AS-IS процесс

![bpmn.as-is.addPatient.svg](..%2FDiagrams%2Fbpmn%2Fbpmn.as-is.addPatient.svg)

**Описание текущего поведения:**

Сотрудник заполняет форму из 5 полей: `name` (одно поле ФИО строкой), `email`, `address` (одна строка), `dateOfBirth`, `registeredDate` (вводится вручную). После сохранения:

1. `patient-service`сохраняет пациента в `patient-service-db`, таблица `patient`(адрес — одна строка, ФИО - одна строка)
2. `billing-service` создаёт биллинговый аккаунт (gRPC, заглушка — ничего не сохраняет)
3. `Kafka` публикует событие `PATIENT_CREATED` в топик `patient` (без настроенного DLQ)
4. `analytics-service` потребляет событие (логирует, без side-effect)
5. Уведомление пациенту **не отправляется**
6. Аккаунт для входа в ЛК **не создаётся**
7. `patient-service` возвращает `frontend` статус-код `200 OK` + `PatientResponseDTO`

**Известные проблемы AS-IS:**

- `name` — одна строка, невозможен корректный поиск по фамилии/имени отдельно (для будущих фич)
- `address` — одна строка, нет нормализации, невозможна аналитика по городам/районам (для будущих фич)
- `registeredDate` передаётся клиентом (некорректно архитектурно)
- `POST /patients` возвращает `200 OK` вместо `201 Created` (некорректно архитектурно)
- `billing-service` не сохраняет данные — (заглушка)
- `notification-service` отсутствует — пациент не получает доступ к ЛК
- `Kafka` (silent data loss): ошибка публикации проглатывается (`log.error` без retry)
- Нет транзакционности — при падении gRPC или Kafka пациент сохранён в БД, но биллинг и уведомление не созданы

## TO-BE процесс

![bpmn.to-be.addPatient.svg](..%2FDiagrams%2Fbpmn%2Fbpmn.to-be.addPatient.svg)

**Описание целевого поведения:**

Сотрудник заполняет форму из **9 полей**: фамилия, имя, отчество (необязательное), email, город, улица, дом, квартира (необязательная), дата рождения. Нажимает кнопку "Сохраненить":

**Синхронная часть (в рамках HTTP запроса):**

1. `patient-service` открывает транзакцию.
2. `patient-service` автоматически проставляет `registeredDate = LocalDate.now()`.
3. `patient-service`сохраняет в `patient-service-db` адрес в таблицу `addresses`, пациента в таблицу `patients` (связь `@OneToOne`).
4. Записывает событие `PATIENT_CREATED` в `patient-service-db` в таблицу `outbox_events` со статусом `PENDING`.
5. Транзакция коммитится — шаги 2, 3, 4 атомарны: либо все вместе, либо откат.
6. `patient-service` возвращает `frontend` статус-код `201 CREATED` + `PatientResponseDTO`.
7. `frontend` закрывает модалку, показывает тост «Пациент успешно добавлен».

    
    После шага 6 HTTP соединение закрыто. Клиент не ждёт выполнения асинхронных шагов ниже.


**Асинхронная часть (вне HTTP запроса):**


8. `OutboxPoller` (@Scheduled каждые 5 сек) читает события со статусом `PENDING`.
9. `patient-service` отправляет событие `PATIENT_CREATED` в `Kafka` топик `patient`. 
10. `patient-service` помечает событие как `PROCESSED` в `outbox_events`.
11. `billing-service` (Kafka consumer) потребляет событие, создаёт биллинговый аккаунт в `billing-service-db`. 
12. `notification-service` (Kafka consumer) потребляет событие, генерирует временный пароль.
13. `notification-service` создаёт аккаунт пациента через gRPC метод `RegisterPatient` в `auth-service`.
14. `notification-service` отправляет пациенту welcome email с логином, паролем и ссылкой на ЛК.
15. `analytics-service` (Kafka consumer) потребляет событие (логирует, без side-effect)

## Общая информация
**Назначение:** Метод предназначен для регистрации нового пациента в системе. Сохраняет данные пациента и адреса в БД, записывает событие в outbox для последующей асинхронной обработки (создание биллингового аккаунта, отправка welcome email).

**Метод:** `POST`

**Внутренний URL:** `http://patient-service:4000/patients`

**Внешний URL:** `http://{gateway-host}/api/patients`

В случае ошибки валидации `GlobalExceptionHandler` перехватывает ошибку и отправляет ответ клиенту `400 BAD REQUEST`.

5. `DispatcherServlet` при успешной валидации вызывает метод `createPatient(patientRequestDTO)` в `PatientController`.

6. `PatientController` вызывает метод `createPatient(patientRequestDTO)` в `PatientService`.

7. `PatientService` выполняет проверку уникальности `email`, вызывая `patientRepository.existsByEmail(patientRequestDTO.getEmail())`. Выполняется SQL-запрос: `SELECT count(*) FROM patients WHERE email = ?`. Если email уже существует — выбрасывается `EmailAlreadyExistsException`, `GlobalExceptionHandler` возвращает клиенту `400 BAD REQUEST, message: "Email address already exists"`.

8. 🟢 `PatientService` открывает транзакцию (`@Transactional`).

9. `PatientMapper` мапит данные из `PatientRequestDTO` в сущности `Patient` и `Address`:
    - 🔴 ~~`patient.name = dto.name`~~
    - 🟢 `patient.lastName = dto.lastName`
    - 🟢 `patient.firstName = dto.firstName`
    - 🟢 `patient.middleName = dto.middleName` (nullable)
    - `patient.email = dto.email`
    - `patient.dateOfBirth = LocalDate.parse(dto.dateOfBirth)` — если строка не соответствует формату `ISO_LOCAL_DATE (YYYY-MM-DD)`, выбрасывается `DateTimeParseException`, транзакция откатывается, клиенту возвращается `500 INTERNAL SERVER ERROR`.
    - 🟢 `patient.registeredDate = LocalDate.now()` — проставляется сервером автоматически.
    - 🟢 `address = new Address(city, street, house, apartment)`
    - 🟢 `patient.address = address`

10. `PatientRepository` сохраняет сущность `Patient` в `patient-service-db`. `Hibernate` выполняет SQL-запросы:
    ```sql
    INSERT INTO addresses (id, city, street, house, apartment) VALUES (?, ?, ?, ?, ?);
    INSERT INTO patients (id, last_name, first_name, middle_name, email, date_of_birth, registered_date, address_id) VALUES (?, ?, ?, ?, ?, ?, ?, ?);
    ```
    🟢 `Address` сохраняется автоматически через `cascade = ALL`.

11. 🟢 `PatientService` записывает событие `PATIENT_CREATED` в таблицу `outbox_events` со статусом `PENDING`:
    ```sql
    INSERT INTO outbox_events (id, patient_id, event_type, payload, status, created_at)
    VALUES (?, ?, 'PATIENT_CREATED', ?, 'PENDING', now());
    ```

12. 🟢 Транзакция коммитится — шаги 9, 10, 11 атомарны. При любой ошибке на этих шагах транзакция откатывается целиком, клиенту возвращается `500 INTERNAL SERVER ERROR`.

13. `PatientMapper` мапит сохранённую сущность `Patient` в объект `PatientResponseDTO`.

14. 🟢 `PatientController` оборачивает `DTO` в `ResponseEntity` со статусом `201 Created` и возвращает в `DispatcherServlet`.

15. `Jackson` выполняет сериализацию объекта в `JSON`.

16. `DispatcherServlet` записывает `JSON` в тело ответа и отправляет клиенту со статусом `201 Created`. HTTP соединение закрывается. Клиент не ждёт выполнения асинхронных шагов.

## Общая информация

**Назначение:** Внутренний компонент `patient-service` обеспечивающий гарантированную доставку событий в Kafka. Читает события со статусом `PENDING` из таблицы `outbox_events`, публикует их в Kafka и помечает как `PROCESSED`. Реализует паттерн Transactional Outbox.

**Тип:** `@Scheduled` компонент

**Класс:** `OutboxPoller`

**Интервал запуска:** каждые 5 секунд (`@Scheduled fixedDelay = 5000`)

**Топик Kafka:** `patient`

---

## Алгоритм работы

1. `OutboxPoller` по расписанию (`@Scheduled fixedDelay = 5000`) вызывает метод `pollOutbox()`.

2. `OutboxPoller` читает из таблицы `outbox_events` все записи со статусом `PENDING`, выполняется SQL-запрос:
    ```sql
    SELECT * FROM outbox_events WHERE status = 'PENDING' ORDER BY created_at ASC;
    ```
   Если записей нет — метод завершается, следующий запуск через 5 секунд.

3. Для каждого события `OutboxPoller` десериализует `payload` из JSON в объект `PatientEvent`.

4. `KafkaProducer` сериализует `PatientEvent` в Protobuf и публикует сообщение в топик `patient`.
    - Если Kafka недоступна или публикация завершилась ошибкой — событие остаётся в статусе `PENDING`, логируется ошибка. Следующая попытка через 5 секунд.

   > **Контракт `PatientEvent` (Protobuf)** описан в разделе [Kafka](#kafka).

5. При успешной публикации `OutboxPoller` обновляет статус события на `PROCESSED` и проставляет `processed_at = now()`:
    ```sql
    UPDATE outbox_events
    SET status = 'PROCESSED', processed_at = now()
    WHERE id = ?;
    ```

6. Шаги 3–5 повторяются для каждой записи из шага 2.
